## 1. 開発背景と意図

### 1.1 解決したい課題

営業・企画の現場では、顧客との商談を進める中で多様な情報が蓄積される。
議事録・メール・資料・会話メモなど、その形式はバラバラで、担当者の頭の中か個人ファイルに散在していることが多い。

結果として以下の問題が生じる：

- **情報の属人化**：担当者が変わると文脈が失われる
- **引き継ぎコスト**：新担当者が状況を把握するまでに時間がかかる
- **文脈の多面性**：同じ案件でも「顧客目線」「営業戦略目線」「実務要件目線」が混在し、どこに何があるか分からない
- **グローバル展開時の言語壁**：日本語で蓄積した情報を英語話者と共有・同期するコストが高い

### 1.2 アプローチ

テキストを AI（Google Gemini）で解析し、**3種の「文脈カード」**として自動分類・整理する。

| カード種別 | 色 | 扱う情報 |
|-----------|---|---------|
| **顧客** | 青 | 顧客企業の状況・課題・意思決定構造・キーパーソン |
| **営業** | 緑 | 自社側の戦略・アプローチ・担当者の方針・社内体制 |
| **業務** | 琥珀 | 具体的な業務要件・成果物・スケジュール・制約 |

この分類は「顧客・自社・実務」という商談の3軸を反映している。複数の視点を混在させず分離することで、情報の検索性・再利用性を高める。

### 1.3 設計思想

- **ローカルファースト**：データはすべてローカル（`db.json`）に保存。SaaS 契約・情報漏洩リスクなし
- **AI はあくまで補助**：AI 抽出結果はドラフト。人間が確認・編集してから保存する
- **プロンプトは開いている**：AI の挙動を制御するプロンプトを設定画面から編集可能。組織のコンテクストに合わせてチューニングできる
- **言語は分離統治**：JA インスタンス・EN インスタンスを独立させ、相互インポートは選択的・手動承認制とする（後述）

## 2. ツール全体像

```
[テキスト入力]（議事録・メモ・資料）
        ↓
  [AI 抽出エンジン]（Gemini 2.0 Flash）
        ↓
[顧客カード] [営業カード] [業務カード]
        ↓ 確認・編集
    [カード一覧]
        ↓ 案件に紐付け
  [案件管理（ケース）]
        ↓ 統合 Export
[Markdown / JSON 出力]（提案書・引き継ぎ資料）
```

## 3. 現バージョンの機能一覧

### 3.1 AI 抽出

#### 汎用抽出（3分類同時）
- テキストを貼り付け、「AIで文脈を抽出」を押すと顧客・営業・業務の3カードを同時生成
- **重点テーマ・絞り込みキーワード**（任意）を指定して抽出方向を誘導可能
  - 例：「CFO が主導する固定費削減施策に注目」

#### 専門抽出（1分類・詳細）
- 対象カテゴリ（顧客 / 営業 / 業務）を選択し、1枚のカードに仔細・網羅的に抽出
- ポリシー文書・業務マニュアルなど情報量が多い資料に有効
- 重点テーマ指定でさらに絞り込み可能

#### ドラフト確認フロー
- 抽出結果は「ドラフト」として表示。タイトル・タグを編集してから保存
- 不要なカードはスキップ可能
- 確認後に「このカードを保存」で DB に登録

### 3.2 カード管理

| 操作 | 説明 |
|------|------|
| 一覧表示 | 全カードをグリッド表示。種別・キーワードでフィルタ・検索 |
| 詳細表示 | カード内容の全文表示、タグ、作成/更新日時 |
| 手動編集 | タイトル・内容・種別・タグを手動編集・保存 |
| AI Refine | 指示テキストを入力して AI でカード内容を改善。適用前に確認可能 |
| 手動作成 | テキスト抽出を使わずカードを新規作成 |
| 削除 | 確認ダイアログ付きで削除 |

### 3.3 案件管理（ケース）

- 複数のカードを1つの「案件」としてグルーピング
- 案件にカードを追加・除外
- 案件内カードの構成を視覚的に確認

### 3.4 統合コンテクスト Export

- 案件に紐付けたカードを AI が統合して Markdown ドキュメントを生成
- 生成結果を確認・手編集してから Markdown / JSON でダウンロード
- エクスポートログ（履歴）を自動保存

### 3.5 プロンプト設定

- AI 抽出・精錬・統合に使うシステムプロンプトを設定画面から編集可能
- 変更はブラウザのローカルストレージに保存（サーバー再起動でリセットされない）
- 「デフォルトに戻す」で初期プロンプトにリセット
- 日本語 / 英語インスタンスで独立したデフォルトプロンプトを持つ

### 3.6 クロスラングインポート（Phase 2 実装済み）

JA インスタンスと EN インスタンスの間で、カード情報を選択的に取り込む機能。

**操作フロー：**

```
カード詳細画面
  ↓ 「他インスタンスから類似カードを検索」
  ↓ AI が類似度をスコアリング（0–100）
候補カード一覧（40% 以上のみ表示）
  ↓ 「差分を分析」
  ↓ Gemini が差分を抽出・目標言語に翻訳
差分コンテンツの確認・編集
  ↓ 選択
    [現在のカードに追記] → カード末尾に追加
    [新規カードとして保存] → 差分を独立カードとして登録
  ↓ インポートログ自動保存
```

**設計方針：**
- インポートは**常に手動確認制**。AI の提案を自動適用しない
- ソース言語のコンテンツは目標言語（JA/EN）に自動翻訳してから提示
- インポートログに「誰が・いつ・どのカードから・どのカードに」取り込んだかを記録

## 4. 画面・操作フロー概要

```
[サイドバー]
├── ＋ 新規抽出       → 抽出ページ
├── カード一覧         → ダッシュボード（全カードグリッド）
│   └── [カードクリック] → カード詳細（編集 / AI Refine / クロスラングインポート）
├── 案件管理           → ケース一覧・詳細・Export
└── プロンプト設定     → プロンプト編集画面
```

### 抽出ページの操作

1. 「汎用抽出（3分類）」または「専門抽出（詳細）」を選択
2. 重点テーマ（任意）を入力
3. テキストを貼り付け
4. 「AIで文脈を抽出する」を押す
5. 各ドラフトを確認し「このカードを保存」または「スキップ」

### カード詳細の操作

1. カード一覧からカードをクリック
2. 右上「編集」で手動編集
3. 「AIでRefine」セクションで改善指示を入力 → 実行 → 確認 → 適用
4. 「クロスラングインポート」セクションで他インスタンスの情報を取り込み

## 5. 日英マルチインスタンス構成（Phase 1/2）

### 5.1 基本構成

本ツールは **JA インスタンスと EN インスタンスを完全分離**して運用する。

| 項目 | JA インスタンス | EN インスタンス |
|------|----------------|----------------|
| 起動コマンド | `npm run dev:ja` | `npm run dev:en` |
| フロント URL | http://localhost:5173 | http://localhost:5174 |
| API ポート | 3001 | 3002 |
| データベース | `db.json` | `db_en.json` |
| UI 言語 | 日本語 | English |
| AI 出力言語 | 日本語 | English |
| デフォルトプロンプト | 日本語版 | English 版 |

### 5.2 データの独立性

- **2つのインスタンスはデータを共有しない**
- JA で作ったカードは JA にのみ存在。EN では別途抽出・作成が必要
- 相互参照・取り込みはクロスラングインポート機能（Phase 2）で手動実施

### 5.3 カノニカル言語の考え方（PoC フェーズ）

現フェーズでは**どちらの言語が「正」かを技術的に強制しない**。運用ルールで管理する想定。

> 💡 **推奨運用（PoC での考え方）**
> - JA が一次情報源（原文・議事録は日本語で管理）
> - EN は JA からの翻訳・派生として位置づける
> - クロスラングインポートで JA → EN に情報を流す方向を基本とする
> - EN 側の編集内容を JA に反映する場合は、EN → JA インポートを手動で実施

### 5.4 Phase 1/2 以降の範囲定義

| フェーズ | 内容 | 状態 |
|---------|------|------|
| Phase 1 | 独立インスタンス（JA/EN）、UI 多言語化、言語別プロンプト | **実装済み** |
| Phase 2 | クロスラングインポート（AI 類似検索・差分翻訳・手動承認） | **実装済み** |
| Phase 3 以降 | ガバナンス強化（詳細は §6 参照） | 次回リリース以降 |

## 6. 次回以降のリリース定義

以下の機能は PoC 検証後、優先度・実現性を評価した上で次回以降のリリースで対応する。

### 6.1 日英ガバナンス（Phase 3）

#### 背景
PoC フェーズでは「誰が何をどのインスタンスに書いていいか」のルールは運用に委ねる。
本番運用に向けては、権限・フロー・整合性の担保が必要になる。

#### 検討中の機能
- **変更通知**：JA カードが更新されたとき、対応 EN カードに「JA が更新されました」バッジを表示
- **同期ステータス管理**：各カードに「JA と同期済み / 未同期 / 差分あり」のステータスを付与
- **承認ワークフロー**：クロスラングインポート時に「提案 → 承認」のステップを追加
- **差分ハイライト**：同期元と現在の内容を並べて差分表示

#### 課題認識
- インスタンス間の「同一カード」の対応付けが複雑（現フェーズはインポートログで追跡）
- 承認者・申請者のユーザー管理が必要になる（→ ユーザー管理機能と連動）

### 6.2 権限設定・ユーザー管理

#### 背景
現バージョンは**シングルユーザー想定のローカルツール**。アクセス制御なし。
複数人・複数組織で使う場合、カード・案件の編集権限・閲覧範囲の制御が必要。

#### 検討中の機能
- ユーザーアカウント（ログイン）
- ロールベースのアクセス制御（管理者 / 編集者 / 閲覧者）
- 案件・カード単位のアクセス権限設定
- 変更履歴の記録と誰が変更したかの追跡

#### 依存関係
- サーバーのマルチユーザー対応（ローカルからクラウド or 認証付きサーバーへの移行）

### 6.3 テキスト以外のコンテンツ取り込み

#### 背景
現バージョンは**テキスト貼り付け専用**。実務では議事録・提案書・カタログなど多様な形式が存在する。

#### 検討中の対応形式

| 形式 | 技術的アプローチ | 優先度（暫定） |
|------|----------------|-------------|
| **PDF** | PDF.js またはサーバー側 pdf-parse でテキスト抽出 | 高 |
| **PowerPoint (.pptx)** | OOXML パース（python-pptx 相当） でスライドテキスト抽出 | 中 |
| **Excel (.xlsx)** | SheetJS でセル値抽出 | 中 |
| **Word (.docx)** | mammoth.js でテキスト変換 | 中 |
| **画像（スキャン・スクリーンショット）** | OCR（Tesseract or Vision API） | 低（精度課題あり） |
| **URL 指定（Web ページ）** | サーバー側でクローリング・本文抽出 | 低（セキュリティ考慮必要） |

#### 設計方針
- ファイルはサーバー側で処理しテキストに変換後、既存の抽出フローに流す
- 抽出元ファイル名・ページ番号をカードのメタデータとして保持
- 画像の OCR は精度が低い可能性があるため、レビュー必須と明示する

### 6.4 その他の検討項目（優先度 TBD）

| 機能 | 概要 |
|------|------|
| **案件テンプレート** | 案件の種類（新規開拓・更新・協業等）に応じたカード構成テンプレート |
| **カード間リンク** | 関連カードへの参照・リンク付け |
| **バージョン履歴** | カード内容の変更履歴・ロールバック |
| **外部連携 Export** | CRM・SFA・Wiki（Notion/Confluence）への出力 |
| **タグ自動補完** | 既存タグのサジェスト |
| **検索強化** | タグ・日付・案件での絞り込み / 全文検索 |
| **クラウド版** | ローカル DB からクラウド DB への移行（チーム共有） |

## 8. 検証項目・手順

### 8.1 前提・環境

| 項目 | 内容 |
|------|------|
| OS | macOS / Linux / Windows（WSL 推奨） |
| Node.js | v18 以上 |
| ブラウザ | Chrome / Edge / Safari（モダンブラウザ） |
| API キー | Gemini API Key（`.env.local` に設定済みであること） |
| 起動方法 | `npm run dev:ja`（JA）/ `npm run dev:en`（EN） |

### 8.2 検証シナリオ一覧

#### シナリオ A：基本抽出フロー（必須）

| # | 操作 | 確認ポイント |
|---|------|------------|
| A-1 | 議事録・商談メモを「テキストを貼り付け」に入力 | テキストの入力・クリアが正常に動作するか |
| A-2 | 「汎用抽出（3分類）」で抽出実行 | 3種類のドラフトが生成されるか |
| A-3 | 各ドラフトの内容を確認 | カード種別（顧客/営業/業務）の分類は適切か |
| A-4 | タイトル・タグを編集して保存 | 保存後にカード一覧に表示されるか |
| A-5 | 特定のカードをスキップ | スキップしたカードは保存されないか |

#### シナリオ B：専門抽出

| # | 操作 | 確認ポイント |
|---|------|------------|
| B-1 | 「専門抽出（詳細）」を選択し、種別を「顧客」に設定 | 顧客カード1枚のみが生成されるか |
| B-2 | 重点テーマを入力して抽出 | テーマに沿った内容が抽出されるか |
| B-3 | 複数の文書を連続で抽出 | 結果の品質・一貫性はどうか |

#### シナリオ C：カード操作

| # | 操作 | 確認ポイント |
|---|------|------------|
| C-1 | カード一覧でキーワード検索 | タイトル・内容・タグでの絞り込みが機能するか |
| C-2 | 種別フィルターで絞り込み | 顧客のみ / 営業のみ 等の表示切替が機能するか |
| C-3 | カード詳細から「AI Refine」を実行 | 指示に沿って内容が改善されるか |
| C-4 | Refine 結果を確認してから「適用」 | 確認なしで書き換えが起きないことを確認 |
| C-5 | カードを手動で新規作成 | 必要項目を入力して保存できるか |
| C-6 | カードを削除 | 削除確認ダイアログが出るか |

#### シナリオ D：案件管理・Export

| # | 操作 | 確認ポイント |
|---|------|------------|
| D-1 | 新規案件を作成しカードを紐付け | 複数種別のカードを同一案件にまとめられるか |
| D-2 | 「AIで統合コンテクストを生成」を実行 | 案件の全カードが統合された Markdown が生成されるか |
| D-3 | 生成結果を確認・手編集してから Download | ファイルがダウンロードされ、内容が正しいか |
| D-4 | エクスポート履歴が保存されるか確認 | 案件画面でログが表示されるか |

#### シナリオ E：プロンプトカスタマイズ

| # | 操作 | 確認ポイント |
|---|------|------------|
| E-1 | 設定画面で「一般抽出 システムプロンプト」を変更して保存 | 変更内容が保存されるか |
| E-2 | 変更後に抽出を実行 | カスタムプロンプトの効果が抽出結果に反映されるか |
| E-3 | 「デフォルトに戻す」を実行 | デフォルトプロンプトに正しく戻るか |

#### シナリオ F：クロスラングインポート（JA ↔ EN）

前提：JA インスタンス（5173）と EN インスタンス（5174）を両方起動していること。

| # | 操作 | 確認ポイント |
|---|------|------------|
| F-1 | JA でカードを作成し、EN で同じ案件の別情報からカードを作成 | 両インスタンスにカードが存在する状態 |
| F-2 | JA のカード詳細で「クロスラングインポート」→「類似カードを検索」 | EN インスタンスに接続し候補が表示されるか |
| F-3 | 類似度スコアは妥当か確認 | 関連するカードに高スコアが付くか |
| F-4 | 候補カードの「差分を分析」を実行 | 差分サマリーと日本語翻訳コンテンツが表示されるか |
| F-5 | 「現在のカードに追記」を実行 | カード末尾に EN 由来情報が追記されるか |
| F-6 | 「新規カードとして保存」を実行 | 差分が独立カードとして作成されるか |
| F-7 | EN → JA 方向でも同様の操作が成立するか確認 | 双方向インポートの動作確認 |

## 9. 既知の制限・注意事項

### 9.1 データの永続化

- データは `db.json`（JA）/ `db_en.json`（EN）にローカル保存される
- **バックアップは手動**。重要なカードを作成したら JSON ファイルのコピーを推奨
- サーバーを終了してもデータは残る

### 9.2 AI の限界

- Gemini API の精度はテキストの品質・量に依存する
- 短すぎる・曖昧なテキストは抽出精度が下がる
- AI の出力は必ず人間がレビューする設計（自動適用しない）

### 9.3 同時アクセス

- **現バージョンは1ユーザーのシングルセッションを想定**
- 同一インスタンスへの複数ブラウザからの同時アクセスはサポート外（データ競合の可能性あり）

### 9.4 Gemini API キー

- API キーは `.env.local` で管理。Git には含まれない
- API の利用量・コストは Google Cloud のダッシュボードで確認すること

### 9.5 クロスラングインポートの前提

- 両インスタンスが起動中であること（片方が停止していると接続エラー）
- 接続エラーは「クロスラングインポート」パネルにエラーメッセージとして表示される
